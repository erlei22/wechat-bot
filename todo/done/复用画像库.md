# 复用现成画像库

先调研有没有成熟的人物画像 / persona 建模库，别从零搭。

## 调研方向

- npm 上的 persona / user-modeling / profile-store 类包。
- 学术圈聊天机器人 persona 的几种主流建模方式（兴趣 / 风格 / 关系图谱）。
- 考虑一下小红书人物画像怎么做的，可以参考
- 隐私和数据归属：能不能本地跑，schema 是否可控。

## 调研结论

**结论：不引入外部库，收敛当前自研 schema。**

### 候选库评估

| 库 | 定位 | 利 | 弊 | 结论 |
|---|---|---|---|---|
| **mem0**（Nous Research 自托管版） | LLM agent 用户记忆管理，向量存储 | 功能完整，支持实体/关系/事实提取，自托管可控 | 依赖向量 DB（重基础设施），为 10-30 人群聊引入成本过高 | ❌ 不引入 |
| **Zep** | 时序知识图谱，企业级 agent 记忆 | 能力强，时序感知 | 更重，需独立服务，schema 不可控 | ❌ 不引入 |
| **persona**（npm） | 身份验证服务（Persona.com）的 JS 客户端 | — | 与用户建模无关，同名不同义 | ❌ 无关 |
| **persona-agent**（GitHub） | AutoGen + MCP 的研究框架 | — | 研究工具，非生产库 | ❌ 无关 |
| 学术库（PersonaMem、DeepPersona 等） | LLM benchmark / 合成画像生成 | — | 研究工具 | ❌ 无关 |

### 小红书画像参考

小红书本质是「兴趣标签（显式：关注/点赞 + 隐式：停留时长/互动）+ 行为标签（发帖频率/类型）」在亿级用户上的大规模计算。我们已经在做轻量版等价实现：LLM 从消息里提取兴趣标签和记录，按群隔离，有操纵检测。差异是量级，不是方法论缺陷，不需要借鉴更复杂的架构。

### 为什么不引入

当前真正的问题是 **schema 没有权威定义**，字段散落在多处字面量中，迭代时容易漏字段或拼写不一致。解法是收敛 schema，不是引入外部库。

### 已做的事

- `profileStore.js` 头部加了权威 schema 注释（字段名、类型、取值范围、约定），明确新字段必须先在此处描述再改其他代码。
- 新增 `createEmptyProfile(name)` 工厂函数，替换三处散落的字面量初始化（`extractAndUpdateProfile` × 1、`extractFromPassiveMessage` × 1），唯一权威的初始化入口。

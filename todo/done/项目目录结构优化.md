# 项目目录结构优化

## 背景

项目从单文件脚本迭代而来，现在功能已经比较完整，但目录层级存在几个明显的混乱点，随着继续加功能会越来越难维护。

## 现状

```
src/
├── adapters/           # lark.js、opencli.js、pi.js — IM 平台 / 外部 CLI 适配器
├── analysis/           # wechatAnalyzer.js — 群聊统计分析
├── config/             # env.js — 全局配置
├── deepseek/           # index.js — DeepSeek API 封装
├── openai/             # index.js — OpenAI API 封装
├── utils/              # replyQueue.js、process.js
├── wechaty/            # sendMessage.js、serve.js、testMessage.js
├── platforms/wechat/   # bot.js、commandRouter.js、botTools.js
│                       # profileStore.js、eventStore.js、eventLifecycle.js
│                       # feedbackStore.js、errorStore.js、messageStore.js
│                       # patternConfig.js、whitelistConfig.js
└── index.js
```

### 具体问题

1. **`wechaty/` 和 `platforms/wechat/` 职责重叠**：`wechaty/sendMessage.js` 是微信消息的分发入口，跟 `platforms/wechat/bot.js` 逻辑上属于同一层，却分散在两个目录，新人很难判断改哪里。

2. **`platforms/wechat/` 过度膨胀**：数据层（store）、业务逻辑（lifecycle）、配置（patternConfig、whitelistConfig）、工具（botTools）全堆在一起，11 个文件没有子分类。

3. **AI provider 各自为目录**：`deepseek/`、`openai/` 各占一个顶级目录，未来加 `claude/`、`ollama/` 还会继续平铺。应收进 `providers/`（或 `llm/`）统一管理。

4. **`adapters/` 和 `platforms/` 语义接近**：`adapters/lark.js` 是飞书适配，`platforms/wechat/` 是微信平台，两者是同级的平台接入层，却放在不同顶级目录。

5. **`analysis/` 是孤立的**：目前只有 `wechatAnalyzer.js`，长期可能挂在某个平台下或归入 `utils`。

## 注意点

- 这是重构，**先改目录再跑测试**，别边改边调 bug。
- `cli.js` 和 `src/index.js` 的入口 import 路径也要一起更新。
- `better-sqlite3` 的 db 路径是运行时拼的（`path.resolve(process.cwd(), dataDir, ...)`），目录移动不影响数据文件，但要确认 import 链没断。
- 建议先出一张「新旧路径对照表」让用户确认，再批量移动，避免返工。
